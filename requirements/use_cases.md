# Use Case Specifications – Space Surveillance Data Pipeline

Спецификации ключевых вариантов использования (формат Cockburn, расширенный):
основной (happy path), альтернативные и исключительные сценарии, пред-/постусловия.

## Акторы

| Актор | Тип | Роль |
|-------|-----|------|
| Заказчик | Первичный, человек/система | Подаёт заявку, получает отчёт |
| Баллистический сервис | Первичный, система | Строит эфемериду, уточняет орбиту |
| Планировщик | Первичный, система | Считает окна видимости, строит график СН |
| Средство наблюдения (СН) | Вторичный, внешняя система | Наводится, снимает, отдаёт сырой поток |
| Конвейер обработки | Первичный, система | Нормализация, валидация, feature extraction |
| ИИ-сервис (Model Serving) | Вторичный, система | Классификация сигнатур (PyTorch/TensorFlow) |
| Аналитик каталога | Вторичный, человек | Верифицирует пограничные случаи, ведёт каталог |

## UC-01 – Принять заявку и построить эфемериду

| Поле | Значение |
|------|----------|
| **ID** | UC-01 |
| **Актор (первичный)** | Заказчик → Баллистический сервис |
| **Цель** | По исходным данным ОН получить эфемериду на интервал |
| **Триггер** | Поступление заявки на наблюдение |
| **Предусловия** | Есть каталожный номер ИЛИ валидный TLE; задан интервал |

### Основной сценарий

1. Заказчик подаёт заявку (каталожный номер/TLE, интервал, приоритет).
2. Система валидирует заявку и извлекает исходные орбитальные данные.
3. Баллистический сервис прогнозирует движение ОН.
4. Формируется эфемерида (координаты + скорость по моментам времени).
5. Заявка переходит в `EPHEMERIS_READY`, эфемерида сохраняется.

### Альтернативные / исключительные сценарии

- **2a. Только TLE без каталожного номера:** эфемерида строится по TLE, объекту
  присваивается временный идентификатор до отождествления.
- **2b. Недостаточно данных:** заявка отклоняется `INSUFFICIENT_ORBITAL_DATA`.

### Постусловия

- **Успех:** эфемерида готова, доступна планировщику.
- **Неуспех:** заявка отклонена, эфемерида не построена.

## UC-02 – Спланировать работу средств наблюдения

| Поле | Значение |
|------|----------|
| **ID** | UC-02 |
| **Актор (первичный)** | Планировщик |
| **Цель** | Назначить окна видимости на инструменты сети без конфликтов |
| **Триггер** | Готовность эфемериды |
| **Предусловия** | Известны параметры пунктов и характеристики 18 СН |

### Основной сценарий

1. Планировщик рассчитывает окна видимости по эфемериде и пунктам.
2. Учитываются высота над горизонтом, освещённость, метео, характеристики СН.
3. По окнам и текущей загрузке строится оптимальный график сети.
4. Каждое окно назначается доступному инструменту, исключая конфликты.
5. График публикуется средствам наблюдения и операторской панели.

### Альтернативные / исключительные сценарии

- **3a. Конфликт за инструмент:** приоритетная заявка получает окно, прочие
  переназначаются на резервные СН или другие окна.
- **1a. Нет окон видимости:** статус `NO_VISIBILITY_WINDOWS`, заказчику
  предлагается расширить интервал.

### Постусловия

- **Успех:** график сформирован, СН получили задания.
- **Неуспех:** наблюдение в интервале невозможно.

## UC-03 – Исполнить наблюдение и собрать сырой поток

| Поле | Значение |
|------|----------|
| **ID** | UC-03 |
| **Актор (первичный)** | Средство наблюдения (СН) |
| **Цель** | Зафиксировать объект и передать сырой поток на обработку |
| **Триггер** | Наступление запланированного окна |
| **Предусловия** | СН исправно и наведено; канал связи доступен |

### Основной сценарий

1. Система выдаёт команду наведения в расчётную точку.
2. СН инициирует съёмку (серия снимков) или радиосеанс.
3. Формируется сырой поток измерений с метками времени и id СН.
4. Поток поступает в систему сбора: Debezium → Kafka → Flink.
5. Flink нормализует и распределяет измерения по сервисам обработки.

### Альтернативные / исключительные сценарии

- **2a. Срыв сеанса (метео/отказ привода):** сеанс `ABORTED`; планировщик
  переназначает окно на резервный инструмент.
- **4a. Перегрузка канала:** Kafka буферизует, обработка идёт по мере готовности
  без потерь (backpressure во Flink).

### Постусловия

- **Успех:** сырые измерения в конвейере, помечены trace_id.
- **Неуспех:** сеанс сорван, инициировано переназначение.

## UC-04 – Предобработать данные и извлечь сигнатуры

| Поле | Значение |
|------|----------|
| **ID** | UC-04 |
| **Актор (первичный)** | Конвейер обработки |
| **Цель** | Из сырых измерений получить чистые сигнатуры объектов |
| **Триггер** | Поступление нормализованных измерений из Flink |
| **Предусловия** | Доступен каталог звёзд для калибровки |

### Основной сценарий

1. Сырые данные проходят нормализацию (устранение инструментальных шумов).
2. Выполняется астрометрическая калибровка по каталогу звёзд.
3. Аномальные измерения отбраковываются по правилам валидации и SLA.
4. Feature extraction выделяет точечные объекты, меряет блеск и координаты.
5. Фоновые звёзды исключаются; формируются сигнатуры объектов.
6. Сигнатуры публикуются в топик `signatures.extracted`.

### Альтернативные / исключительные сценарии

- **2a. Кадр не прошёл контроль качества:** `QUALITY_FAILED`, исключение из
  анализа, инцидент.
- **4a. Перекрытие объектов на кадре:** применяется разделение источников (PSF-fit).

### Постусловия

- **Успех:** сигнатуры готовы к ИИ-анализу.
- **Неуспех:** кадр отбракован.

## UC-05 – Проанализировать сигнатуру ИИ-моделями (ключевой)

| Поле | Значение |
|------|----------|
| **ID** | UC-05 |
| **Актор (первичный)** | Конвейер обработки → ИИ-сервис (Model Serving) |
| **Цель** | Классифицировать сигнатуру: известный / новый / аномалия |
| **Триггер** | Событие `signatures.extracted` |
| **Предусловия** | Загружена актуальная версия ИИ-модели; доступны эталоны каталога |

### Основной сценарий

1. Сигнатура подаётся в ИИ-сервис (PyTorch/TensorFlow) на инференс.
2. Модель сравнивает сигнатуру с эталонами каталога.
3. При совпадении выше порога объект помечается `IDENTIFIED` (caталожный id + confidence).
4. Результат публикуется в `ai.classifications`, передаётся на уточнение орбиты.

### Альтернативные сценарии

- **2a. Нет совпадения:** сигнатура → `NEW_OBJECT`, инициируется заведение ОН (UC-06).
- **2b. Аномальное поведение известного ОН:** метка `ANOMALY` (тип + confidence),
  формируется приоритетное оповещение.
- **3a. Пограничная уверенность:** результат ниже порога автоматического решения –
  сигнатура направляется Аналитику (human-in-the-loop).

### Исключительные сценарии

- **1a. ИИ-сервис недоступен / таймаут (Circuit Breaker = OPEN):** сигнатура
  буферизуется в очередь, анализ повторяется после восстановления; потерь нет.
- **1b. Несовместимая версия модели:** инференс отклоняется, событие эскалируется
  на ML-эксплуатацию.

### Постусловия

- **Успех:** сигнатура классифицирована, результат опубликован.
- **Отложено:** сигнатура в очереди на повторный инференс.

## UC-06 – Уточнить орбиту и обновить каталог

| Поле | Значение |
|------|----------|
| **ID** | UC-06 |
| **Актор (первичный)** | Баллистический сервис → Каталог |
| **Цель** | Сохранить уточнённую орбиту с версионированием и lineage |
| **Триггер** | Результат ИИ-классификации (`IDENTIFIED`/`NEW_OBJECT`/`ANOMALY`) |
| **Предусловия** | Доступен Data Lake (PostgreSQL + S3 + Iceberg + Atlas) |

### Основной сценарий

1. По новым измерениям уточняется вектор состояния (орбита) идентифицированного ОН.
2. Пересчитывается доверительная область орбиты.
3. Изменения сохраняются в Data Lake с новой версией схемы/таблицы.
4. Полный lineage измерения фиксируется в Apache Atlas.
5. Публикуется событие `catalog.updated`.

### Альтернативные / исключительные сценарии

- **1a. Новый объект:** заводится запись с первоначальной орбитой и доверительной
  областью, присваивается постоянный каталожный id.
- **1b. Аномалия:** объект помечается, к записи прикладывается тип аномалии и
  оповещение.

### Постусловия

- **Успех:** каталог обновлён, версия и lineage зафиксированы.
