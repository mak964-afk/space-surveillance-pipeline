# User Stories – Space Surveillance Data Pipeline

Формат: **Connextra** (`As a … I want … so that …`) + **Acceptance Criteria** в
нотации **Gherkin** (Given–When–Then). Приоритет – MoSCoW, оценка – story points.

Акторы: **Заказчик**, **Баллистик**, **Оператор сети наблюдения**, **Средство
наблюдения (СН)** (внешняя система), **ИИ-сервис** (Model Serving),
**Планировщик** (системный), **Аналитик каталога**.

---

## US-01 – Приём заявки и построение эфемериды

**Приоритет:** Must · **SP:** 8

> **Как** заказчик,
> **я хочу** подать заявку на наблюдение объекта по его каталожному номеру или TLE,
> **чтобы** получить актуальные данные о его положении и состоянии.

### Acceptance Criteria

```gherkin
Feature: Заявка и эфемерида

  Background:
    Given заказчик подаёт заявку на объект с каталожным номером "25544"
    And указывает требуемый интервал наблюдения

  Scenario: Построение эфемериды по каталожному номеру
    When система принимает заявку
    Then по каталожному номеру извлекаются исходные орбитальные данные
    And баллистический сервис строит эфемериду на запрошенный интервал
    And эфемерида содержит координаты и скорость на последовательные моменты времени
    And заявка переходит в статус "EPHEMERIS_READY"

  Scenario: Заявка по TLE без каталожного номера
    Given заказчик передал элементы орбиты в формате TLE
    When система принимает заявку
    Then эфемерида строится непосредственно по TLE
    And объекту присваивается временный идентификатор до отождествления

  Scenario: Недостаточные исходные данные
    Given в заявке нет ни каталожного номера, ни валидного TLE
    When система валидирует заявку
    Then заявка отклоняется с кодом "INSUFFICIENT_ORBITAL_DATA"
    And эфемерида не строится
```

## US-02 – Планирование работы средств наблюдения

**Приоритет:** Must · **SP:** 13

> **Как** оператор сети наблюдения,
> **я хочу** автоматически получать оптимальный график работы 18 инструментов на
> основе окон видимости,
> **чтобы** минимизировать простои и максимально использовать сеть.

### Acceptance Criteria

```gherkin
Feature: Планирование наблюдений

  Scenario: Расчёт окон видимости
    Given построена эфемерида объекта
    When система рассчитывает условия видимости для пунктов наблюдения
    Then учитываются координаты пункта, минимальная высота над горизонтом,
      условия освещённости, метеоограничения и характеристики СН
    And формируется список окон видимости с привязкой к конкретным СН

  Scenario: Оптимальное распределение по сети из 18 инструментов
    Given рассчитаны окна видимости и известна текущая загрузка СН
    When микросервис планирования составляет график
    Then каждое окно назначается доступному инструменту без конфликтов
    And расписание минимизирует простои инструментов
    And приоритетные заявки получают окна в первую очередь

  Scenario: Нет доступных окон в требуемом интервале
    Given объект не поднимается над минимальной высотой ни на одном пункте
    When система планирует наблюдение
    Then возвращается статус "NO_VISIBILITY_WINDOWS"
    And заказчику предлагается расширить интервал наблюдения
```

## US-03 – Исполнение наблюдения и сбор сырого потока

**Приоритет:** Must · **SP:** 13

> **Как** средство наблюдения,
> **я хочу** получить команду наведения в расчётное время и передать сырой поток
> измерений,
> **чтобы** зафиксировать объект и направить данные на обработку.

### Acceptance Criteria

```gherkin
Feature: Исполнение наблюдения

  Scenario: Наведение и съёмка в расчётное окно
    Given наступило запланированное окно наблюдения для СН
    When система выдаёт команду наведения в расчётную точку небесной сферы
    Then инициируется съёмка (серия снимков) или радиолокационный сеанс
    And формируется сырой поток данных с метками времени и идентификатором СН

  Scenario: Потоковый приём сырых данных
    Given СН передаёт сырой поток по защищённому каналу
    When поток поступает в центральную систему сбора
    Then Debezium фиксирует изменения, Kafka буферизует, Flink нормализует поток
    And каждое измерение получает trace_id для сквозной прослеживаемости

  Scenario: Срыв сеанса наблюдения
    Given во время окна СН недоступно (метео, отказ привода)
    When сеанс срывается
    Then сеанс помечается статусом "ABORTED" с причиной
    And планировщик пытается переназначить окно на резервный инструмент
```

## US-04 – Предобработка и извлечение сигнатур

**Приоритет:** Must · **SP:** 13

> **Как** аналитик каталога,
> **я хочу** чтобы сырые измерения автоматически нормализовались и из них
> извлекались сигнатуры объектов,
> **чтобы** ИИ-анализ работал на чистых компактных данных, а не на «шуме».

### Acceptance Criteria

```gherkin
Feature: Предобработка и feature extraction

  Scenario: Нормализация и калибровка оптических данных
    Given поступила серия сырых снимков участка неба
    When данные проходят слой нормализации
    Then устраняются инструментальные шумы
    And выполняется астрометрическая калибровка по каталогу звёзд
    And явно аномальные измерения отбраковываются по правилам валидации и SLA

  Scenario: Извлечение сигнатур из снимка
    Given снимок нормализован и откалиброван
    When выполняется feature extraction
    Then на снимке выделяются все точечные объекты
    And для каждого измеряется блеск (фотометрия) и координаты (астрометрия)
    And фоновые звёзды исключаются сравнением с каталогом
    And по каждому объекту формируется сигнатура

  Scenario: Кадр не прошёл контроль качества
    Given калибровка снимка не достигла требуемой точности
    When контроль качества оценивает кадр
    Then кадр помечается "QUALITY_FAILED" и исключается из анализа
    And инцидент фиксируется для разбора
```

## US-05 – ИИ-анализ и классификация сигнатур

**Приоритет:** Must · **SP:** 21

> **Как** аналитик каталога,
> **я хочу** чтобы извлечённые сигнатуры автоматически классифицировались
> ИИ-моделями (известный / новый / аномалия),
> **чтобы** отождествление и обнаружение шли без ручного перебора каталога и в
> темпе поступления данных.

> Это центральная история конвейера – именно здесь ИИ создаёт основную ценность.

### Acceptance Criteria

```gherkin
Feature: ИИ-анализ и классификация

  Scenario: Идентификация известного объекта
    Given готова сигнатура наблюдаемого объекта
    When сигнатура подаётся в ИИ-сервис (PyTorch/TensorFlow)
    Then модель сравнивает её с эталонами каталога
    And при совпадении выше порога объект помечается "IDENTIFIED"
    And возвращается каталожный идентификатор и confidence-оценка

  Scenario: Обнаружение нового объекта
    Given сигнатура не совпадает ни с одним эталоном каталога
    When ИИ-сервис классифицирует сигнатуру
    Then она помечается "NEW_OBJECT"
    And инициируется заведение нового ОН с первичной орбитой

  Scenario: Детекция аномалии известного объекта
    Given объект идентифицирован, но его поведение отличается от ожидаемого
    When ИИ-модель оценивает признаки (блеск, манёвр, фрагментация)
    Then выставляется метка "ANOMALY" с типом аномалии и confidence
    And формируется приоритетное оповещение заказчику

  Scenario: Недоступность ИИ-сервиса – graceful degradation
    Given ИИ-сервис (model serving) недоступен или превысил таймаут
    When сигнатура направляется на инференс
    Then срабатывает Circuit Breaker, сигнатура буферизуется в очередь
    And анализ повторяется после восстановления сервиса
    And ни одна сигнатура не теряется

  Scenario: Пограничная уверенность – human-in-the-loop
    Given confidence классификации ниже порога автоматического решения
    When ИИ возвращает неуверенный результат
    Then сигнатура направляется аналитику на ручную верификацию
    And решение аналитика используется для дообучения (вне scope данного сервиса)
```

## US-06 – Уточнение орбиты и обновление каталога

**Приоритет:** Must · **SP:** 13

> **Как** аналитик каталога,
> **я хочу** чтобы новые измерения уточняли орбиту объекта и версионируемо
> сохранялись в каталоге,
> **чтобы** данные были точными и полностью прослеживаемыми.

### Acceptance Criteria

```gherkin
Feature: Каталог и уточнение орбиты

  Scenario: Уточнение вектора состояния известного объекта
    Given объект идентифицирован, получены новые измерения
    When выполняется уточнение орбиты
    Then вектор состояния пересчитывается по треку (или серии разновременных треков)
    And обновляется доверительная область орбиты

  Scenario: Заведение нового объекта в каталог
    Given ИИ пометил сигнатуру как "NEW_OBJECT"
    When создаётся запись в каталоге
    Then рассчитывается первоначальная орбита и доверительная область
    And объекту присваивается постоянный каталожный идентификатор

  Scenario: Версионирование и аудит данных
    Given изменения вносятся в каталог
    When данные сохраняются в Data Lake (PostgreSQL + S3 + Iceberg)
    Then фиксируется новая версия схемы/таблицы
    And полный lineage измерения прослеживается через Apache Atlas
```

## US-07 – Формирование и доставка отчёта

**Приоритет:** Should · **SP:** 8

> **Как** заказчик,
> **я хочу** получить итоговый отчёт по заявке в согласованном формате,
> **чтобы** использовать результаты наблюдения в своих задачах.

### Acceptance Criteria

```gherkin
Feature: Отчёт заказчику

  Scenario: Сборка и доставка отчёта
    Given все этапы обработки по заявке завершены
    When система комплектует отчёт
    Then отчёт содержит идентификацию объекта (или статус «новый»),
      уточнённые орбитальные параметры, метки времени наблюдений,
      использованные СН и оценку точности измерений
    And отчёт формируется в согласованном формате
    And доставляется заказчику по защищённому каналу

  Scenario: Приоритетный отчёт по аномалии
    Given по объекту выявлена аномалия
    When формируется отчёт
    Then отчёт помечается как приоритетный
    And доставляется заказчику в ускоренном режиме
```

## Нефункциональные требования (NFR)

| ID | Категория | Требование |
|------|-----------|------------|
| NFR-01 | Производительность | Время «съёмка → отчёт» ≤ 20 минут (p95) |
| NFR-02 | Пропускная способность | Потоковая обработка измерений со всех 18 инструментов без потерь |
| NFR-03 | Отказоустойчивость ИИ | Circuit Breaker + Retry на model serving; сигнатуры не теряются |
| NFR-04 | Надёжность стриминга | At-least-once (Debezium→Kafka→Flink); дедуп по (sensor_id, frame_id) |
| NFR-05 | Прослеживаемость | 100 % измерений с полным lineage (Apache Atlas) |
| NFR-06 | Версионирование данных | Iceberg-версии схем/таблиц каталога, аудит изменений |
| NFR-07 | Наблюдаемость | Сквозной trace_id от заявки до отчёта; операторские панели реального времени |
| NFR-08 | Безопасность | Защищённые каналы передачи; разграничение доступа; деперсонализация артефактов |
| NFR-09 | Качество данных | Контроль качества кадров по правилам валидации и SLA до ИИ-анализа |
